## §5. 予測モデル

### §5.1 符号規約

- **収支 d = 消費kcal − 摂取kcal**。d > 0（黒字）はカロリーを削減できた日＝減量に有利、d < 0（赤字）はオーバー。
- **累積収支 cum** = その期間の d の累積和。
- **予実乖離** = 実測体重 − 予測体重。正なら「予測より重い＝ペースが遅れている」、負なら「予測より軽い＝先行」。
- 実装: `lib/data.ts`（`DayData.d` / `DayData.cum`）。予実乖離の数値計算は `lib/advisor.ts`（`buildAdvicePrompt` 内、AIアドバイザーへのプロンプト生成時に算出）で行う。`components/ForecastTab.tsx` の `TrajectoryChart` は実測/予測/目標の3本の線を視覚的に比較表示するのみで、乖離の数値自体は計算・表示していない。

### §5.2 個人係数 k の算出方法

個人係数 k（kcal/kg）は「体重を 1kg 動かすのに必要なカロリー収支」を表す。理論値（脂肪組織の熱量）は約 7,700 kcal/kg だが、実際の体組成変化（水分・筋肉量を含む）は個人差が大きいため、本アプリは既定値 **k₀ = 7,200** から出発し、実測データが十分蓄積した段階で個人ごとに回帰キャリブレーションする。

#### §5.2.1 現行実装（v1.0）— 最小二乗回帰 + 閾値ゲート

**実装箇所**: `app/page.tsx`（回帰計算・キャリブレーション判定）、`components/ForecastTab.tsx`（散布図表示・期間参考値）。

**入力データ**: 体重が実測されている日（`w > 0`）の `(cum, w)` ペア（累積収支・体重）。

**回帰**: 単純最小二乗（OLS）で `w = slope × cum + intercept` を推定し、`k_raw = −1 / slope` とする。

**キャリブレーション条件**（すべて満たす場合のみ実測値を採用、それ以外は既定値 7,200 を使用）:
| 条件 | しきい値 |
|---|---|
| 測定日数 | ≥ 10日 |
| 測定期間の幅 | ≥ 21日（暦日、連続でなくてよい） |
| 算出値の妥当域 | 4,000 ≤ k_raw ≤ 12,000 kcal/kg |

採用時は `k = round(k_raw / 50) × 50`（50kcal刻みに丸め）。

**既知の弱点**:
- OLS は外れ値（測定誤差・水分変動による体重の急な跳ね）に弱く、少数の外れ値で傾き（＝k）が大きく歪みうる。
- 体重の生値をそのまま回帰に使っており、日々のノイズ（水分・食事・測定タイミングで ±1〜2kg）を平滑化していない。
- 表示期間を絞った際（`ForecastTab` の「体重×累積黒字」カード、10日/30日窓）は測定点数が減り、上記の弱点がより顕著に出る。この期間別kは**参考値表示のみ**で、予測（トラジェクトリ・ホーム）には全期間の較正済みkを使用する（`ForecastTab.tsx` の `windowK` / `kVal` の分離を参照）。
  - 補足: この期間別参考値（`windowK`）は、全期間の較正済みk（`kVal`）が課す日数・期間ゲート（≥10日・≥21日）を課さず、妥当域チェック（4,000〜12,000）のみで算出される。短期間窓ほど、このチェックだけでは不安定な値が出やすい。

#### §5.2.2 改訂方針（v2.0）— ロバスト統計手法への置き換え

v1.0 の弱点（外れ値脆弱性・ノイズ未処理）に対応するため、体重時系列の解析で確立された手法に置き換える。個々の手法は独立して採用実績があり、組み合わせて使うことで日次ノイズと外れ値の双方に頑健なトレンド・傾き推定を行う。

> **本節のステータス**: 定量検証済み（§5.2.3参照）。結論として、**v2.0案は支配的な誤差要因（摂取側の系統的過小申告バイアス）を補正できないことが判明**した。以下のパイプライン記述は設計方針の記録として残すが、この形での実装は推奨しない。§5.2.3の結論・代替案を参照。

**設計方針の出典・根拠**:
1. **The Hacker's Diet**（John Walker）が提唱し、TrendWeight・Libra・Happy Scale 等の体重トラッキングアプリが採用する **EWMA（指数加重移動平均）** による体重トレンド抽出。日々の水分変動を均し、脂肪トレンドのみを残す。
   出典: [The Hacker's Diet — Signal and Noise](https://www.fourmilab.ch/hackdiet/e4/signalnoise.html), [TrendWeight Help/FAQ](https://trendweight.com/help/)
2. **Theil–Sen 推定量**（全点ペアの傾きの中央値）による傾き推定。破綻点（breakdown point）約29.3%で、外れ値に対して最小二乗より大幅に頑健。線形トレンド推定で最も広く使われるノンパラメトリック手法とされる。
   出典: [Theil–Sen estimator (Wikipedia)](https://en.wikipedia.org/wiki/Theil%E2%80%93Sen_estimator), [The Theil-Sen robust estimator (SAS Blog)](https://blogs.sas.com/content/iml/2019/05/28/theil-sen-robust-regression.html)
3. **Hampel フィルタ / MAD（中央絶対偏差）** による外れ値検出。移動窓内の中央値からの乖離を MAD でスケールし、しきい値（一般に 3×MAD 相当）を超える点を外れ値として棄却・置換する。平均・標準偏差ベースの手法と異なり、外れ値自体による汚染を受けにくい。
   出典: [Outlier Detection with Hampel Filter (Towards Data Science)](https://towardsdatascience.com/outlier-detection-with-hampel-filter-85ddf523c73d/), [A Novel Approach to Speed Up Hampel Filter for Outlier Detection (MDPI Sensors, 2025)](https://www.mdpi.com/1424-8220/25/11/3319)
   - 補足: Hampel/MAD（②点レベルの外れ値除去）と Theil-Sen（④傾きのロバスト推定）は役割が異なり、組み合わせても冗長ではない。ただし③のEWMA平滑化は②で除去しきれなかった外れ値の影響を近傍点に拡散させうるため、④が前提とする「外れ値は孤立点」という仮定を弱める可能性がある（③→④の順序に伴う既知の論点、後述）。

**処理パイプライン**（体重時系列 → k）:

```
生の体重時系列
  → ① 生理的妥当性クリップ（前日比の異常な変化を除外・クリップ）
  → ② Hampel/MAD による外れ値検出・置換（移動窓の中央値 ± n×MAD）
  → ③ EWMA 平滑化（α ≈ 0.1、日々の水分変動を均す）
  → ④ Theil-Sen ロバスト回帰（累積収支 vs 平滑化体重の傾き）
  → ⑤ 妥当域クランプ + 前回値フォールバック
  → k
```

**① 生理的妥当性クリップ**: 1日あたりの体重変化が生理的に妥当な範囲を超える場合（水分/測定誤差の可能性が高い）、その日の変化量を上限にクリップする。

**② 外れ値検出（Hampel/MAD）**: 移動窓（例: 前後数日）内で中央値と MAD を算出し、`|値 − 中央値| > n × MAD × 1.4826` の点を外れ値として中央値で置換する。

**③ EWMA平滑化**: `trend[t] = trend[t-1] + α × (raw[t] − trend[t-1])`。α が小さいほど平滑（ノイズ除去↑・追従性↓）。平滑化後の系列を「トレンド体重」として以降の回帰に使う。

**④ Theil-Sen 回帰**: トレンド体重と累積収支の全点ペアの傾きの中央値を採用し、`k_raw = −1 / median(slope)` を算出。
   - **既知の論点**: ③でy軸（体重）のみをEWMA平滑化し、x軸（累積収支）は未平滑のまま回帰している。EWMAは入力に対して遅延（ラグ）を持つため、ダイエット開始/停止直後など収支ペースが非定常な時期に傾き推定が偏る可能性がある。両軸の平滑化を揃えるか、非対称にする理由を明確化する必要がある（未検証）。

**⑤ 妥当域クランプとフォールバック**:
- 算出値が妥当域（v1.0 と同じ 4,000〜12,000 kcal/kg）を外れた場合、または収支と体重変化の**符号が矛盾する**場合は、**前回採用値（`kPrev`）を維持**する（既定値へ直接落とすのではなく、直前の安定値を保つ）。
  - 「符号矛盾」の具体的な判定条件（何と何を比較し、どの閾値を超えたら矛盾とみなすか）は**未確定**。妥当域クランプ（4,000〜12,000）が既に負・非有限の傾きを排除するため、これが追加でどの異常系を捕捉する想定かも含めて設計時に定義する。
- 初回（`kPrev` 未設定）は既定値 7,200 にフォールバック。
- 体重がほぼ動いていない場合（変化が閾値未満）も `kPrev` を維持し、ノイズによる誤ったk更新を防ぐ。閾値の具体値は**未確定**。

**可変窓（ハイブリッドウィンドウ）**: 直近の測定密度に応じて回帰に使う窓の長さを調整する（データが疎な期間は窓を広げて点数を確保し、密な期間は直近に絞ってトレンドの鮮度を保つ）。窓の上限は56日程度を想定。v1.0のゲート（測定日数≥10・期間≥21日）は本パイプラインでも**下限の前提条件として暫定的に踏襲**する（可変窓がこの下限を緩和・置換するものではない）。密度に応じた具体的な窓調整式は**未確定**。

**診断シグナル**: 算出結果に「クランプが発生したか」「符号矛盾が検出されたか」を示すフラグと理由を付随させる。これは UI 上のエラー表示には使わず、ログおよび AI アドバイザー（`lib/advisor.ts` の advice 生成プロンプト）への文脈情報として扱う——「モデル破綻の兆候」を検知した旨をアドバイザーの分析材料に含めることで、より的確な原因説明（記録漏れの疑い等）に繋げる。フラグを `lib/advisor.ts` の `AdviceContext` にどう統合するかは**未確定**（現状同インターフェースに該当フィールドは無い）。

**未確定パラメータ・設計事項一覧**（実装着手前に確定が必要）:

| 項目 | 内容 |
|---|---|
| Hampel窓幅・MAD倍率n | 「前後数日」「一般に3×MAD相当」としか記載がなく未確定 |
| 生理的クリップの上限値 | 1日あたり何kgまで許容するか未確定 |
| EWMAのα | 「≈0.1」のみで確定値なし |
| 符号矛盾の判定条件 | 比較対象・閾値とも未定義 |
| 「体重がほぼ動いていない」閾値 | 未確定 |
| 可変窓の下限・密度依存の調整式 | 上限(56日)のみ規定、具体式なし |
| 診断フラグのスキーマ・advisor.ts統合方法 | 未確定 |

**移行時の非互換点**（v1.0 → v2.0 実装時に判断が必要な事項）:
- v1.0 は k をクライアント（`app/page.tsx`）で全データ再計算しているが、v2.0 は EWMA の逐次更新・前回値保持（`kPrev`）を要するため、**サーバ側で永続化した状態**（前回採用k・更新時刻・診断フラグ）を持つ設計に変える必要がある。
- 永続化が必要な項目: 前回採用k値、最終更新日時、算出に使った窓の日数（診断用）、クランプ/符号矛盾の発生有無とその理由（診断用）。
- 上記はユーザー単位で保持する（マルチユーザー対応後は `user_id` スコープ必須）。

#### §5.2.3 定量検証（v1.0/v2.0案の精度評価）

v1.0（現行）と v2.0（draft）が、このアプリの実運用データ（自己申告の食事記録・ウェアラブル由来の消費カロリー推定・家庭用体重計の実測）を前提にどの程度の精度を出せるかを、モンテカルロシミュレーション（Node.js、真値k=7,700、週0.5kg減想定、試行数2,000〜3,000）で検証した。

**最重要の結論**: k推定誤差の支配的な要因は**ランダムノイズではなく、摂取カロリー自己申告の系統的な過小申告バイアス**である。v2.0案（EWMA・Hampel/MAD・Theil-Sen）はすべて**体重（y軸）側**のクリーニングであり、**累積収支（x軸）側の系統誤差には一切効かない**。

**バイアスの解析的な導出**: 摂取が真値の割合 u だけ過小申告されると、測定累積収支は真の収支より速く積み上がる（`cum_meas` が `≈u×平均摂取×日数` だけ過大評価）。`k_raw = −1/slope` のため、これは**窓の長さや推定手法のロバスト性によらない一定のバイアス**（`Δk ≈ u×平均摂取/目標減量ペース`）を生む。

**シミュレーション結果**（消費ノイズ12%・摂取記録ノイズ15%・体重ノイズ±0.7kg・8%の確率で±1.5〜3kgの体重外れ値、を付加）:

| シナリオ | 90日窓 | ゲート通過率 |
|---|---|---|
| ノイズのみ（過小申告なし） | 平均7,762・sd784（RSD≈10%） | 100% |
| 摂取10%過小申告 | 平均10,744・sd860 | **93%が通過**（≈+39%の誤差なのに「較正成功」と判定される） |
| 摂取20%過小申告 | 平均13,772（理論値13,720と一致） | ~3%（妥当域12,000超で偶然弾かれる） |
| 摂取15%過小申告 | 平均≈12,300 | ~39%通過 |

ノイズのみなら60〜90日で収束するが（v1.0のゲートは有効に機能）、**10〜15%という現実的な過小申告率**では、ゲートをすり抜けて「較正済み」に見える大きく誤ったkが採用される。

**Theil-Sen は改善しない、むしろ悪化しうる**: ノイズのみ・30日窓では OLSのsd=2,579 に対し Theil-Sen は sd=293,423（xにもノイズがあると全点ペア中央値法が不安定化）。過小申告バイアスに対しても Theil-Sen は OLS と同等かそれ以上の誤差（20%過小申告・90日: OLS 13,772 vs Theil-Sen 14,423）— **系統バイアスへの補正効果はゼロ**。

**ユーザー影響の実感値**（現在体重−目標=8kg、残り86日、真のk=7,700→目標日次黒字716kcal）:

| 採用されたk | 目標摂取カロリーのズレ | 90日後体重予測のズレ |
|---|---|---|
| 8,500（+10%誤差） | +74kcal | +0.61kg |
| 10,000（現実的な過小申告レベル） | **+214kcal（+30%）** | **+1.48kg** |
| 12,000（妥当域上限） | +400kcal（+56%） | +2.30kg |

**副次的に発見した実装バグ（修正済み）**: `app/page.tsx` の `dailyTarget`（ホーム画面の目標カロリー・オントラック判定の元値）が、較正済みの `kVal` ではなく**リテラルの `7200` を直接参照**していた。つまり従来はキャリブレーションの結果がホーム画面の表示に一切反映されず、`ForecastTab` の予測線や `lib/advisor.ts` のAIアドバイス内部計算とは異なる値を表示しうる状態だった。`kVal` の算出を `dailyTarget` の前に並べ替え、`kVal` を参照するよう修正した（本コミットで対応）。

**結論**:
- **v1.0は不十分**: ゲート（10日/21日/4,000-12,000）は大きなバイアスは弾けるが、現実的な10〜15%の過小申告を「較正成功」として通してしまう構造的な盲点がある。
- **v2.0案（draft）も的外れ**: 体重側だけを頑健化しても、支配的な誤差要因（収支側の系統バイアス）には無力。このままの形での実装は推奨しない。
- **より効くレバー（今後の検討候補）**: 自己申告の摂取量を独立推定TDEE（活動量から推定される消費カロリーの目安）と突き合わせて過小申告を検知・補正する、または累積収支への依存を減らし体重トレンドの時間変化から直接ペースを推定する設計。いずれも本ドキュメントでは未設計（別途検討が必要）。

#### §5.2.4 採用設計（v3.0）— アダプティブTDEE + 体重トレンドのみの軌跡予測

§5.2.3の結論を受け、k算出ロジックを全面的に再設計した。**実装済み**（`lib/forecast.ts`）。新規データ収集は行わない（Google Healthは`total-calories`という単一値のみを提供しBMR/活動の分離ができず、体組成データも体重と独立して疎なため、これらを主要シグナルにする代替案は現状データでは実現不可と判明済み）。

**設計の骨子**: 「個人係数kを回帰で較正する」というアプローチ自体を放棄する。kは**固定の物理定数**（`K_CONST = 7200`、変更なし）に戻し、代わりに以下の2つの独立した変更を行う。

**変更A（軌跡予測）**: 予測体重の算出を「体重 vs 累積申告収支」の回帰から、「体重 vs 時間（カレンダー日）」のみの回帰に変更した。外れ値除去・EWMA平滑化・Theil-Senロバスト回帰という統計パイプラインはv2.0案から流用しているが、**適用先を正しい軸（体重×時間、双方ともバイアスを持たない）に変えた**。§5.2.3で指摘した「v2.0の統計手法は体重側だけを頑健化しても収支側の系統バイアスには無力」という問題は、**そもそも累積収支という軸自体を使わなくなったことで構造的に解消**される。

**変更B（目標摂取カロリー）**: 「実際の体重変化」（変更Aと同じトレンド推定）と「本人の申告摂取量」から、その人の"実効TDEE"を、**本人が申告する単位系のまま**逆算する（アダプティブTDEE）:

```
avgTDEE = avgLoggedIntake − (体重トレンドの変化量 × K_CONST / 期間日数)
目標摂取カロリー = avgTDEE − 目標欠損((curW−tgtW)×K_CONST/残り日数)
```

申告の過小申告率が期間内でおおむね一定なら、`avgTDEE`にも同じ分だけの歪みが乗るため、最終的な目標摂取カロリーの提示ではその歪みが**自己相殺される**（両方とも「本人が実際に申告する単位」で表現されているため）。

**v2.0案の2つの技術的懸念の解消**:
- 「EWMAラグ不整合」（§5.2.2 既知の論点④）：もはや累積収支（x軸）が存在せず、時間軸は測定誤差を持たないため、この問題自体が発生しない。
- 「サーバ側`kPrev`永続化」（§5.2.2 移行時の非互換点）：kは固定定数となり較正・前回値保持の概念自体が無くなったため、DBスキーマ変更・マイグレーションは一切不要（`lib/forecast.ts`はステートレスで、保存済み`DayData[]`から毎回再計算する）。

**アルゴリズムの確定パラメータ**（§5.2.2「未確定パラメータ一覧」の解消状況）:

| §5.2.2での未確定項目 | この設計での解決 |
|---|---|
| Hampel窓幅・MAD倍率n | 窓5点・n=3×MAD、5点未満は外れ値判定をスキップ、で確定 |
| 生理的クリップの上限値 | 1.5kg/日（測定間隔日数に比例して緩和）で確定 |
| EWMAのα | 0.1で確定 |
| 符号矛盾の判定条件 | **概念自体を廃止**（kPrevが存在しないため） |
| 「体重がほぼ動いていない」閾値 | **概念自体を廃止**（同上） |
| 可変窓の下限・密度依存の調整式 | Theil-Senは全点ペア中央値（365点上限で最大約6.6万ペア、同期実行可）。念のため400点超で均等間引きする防御的キャップのみ設定。可変窓という概念は採用せず、固定窓（`TDEE_WINDOW_DAYS=21`）+ 全期間トレンドの2種を用途別に使い分ける形にした |
| 診断フラグのスキーマ・advisor.ts統合方法 | クランプ/kPrevの概念が無いため診断フラグは不要に。代わりに`TdeeSource`（`adaptive`/`avgBurnFallback`/`insufficientData`）で状態を表現し、`lib/advisor.ts`のプロンプトに直接反映 |

**信頼度は情報表示のみ（もうゲートではない）**: v1.0は「較正成功/失敗」の二択でロジックそのものを分岐させていたが、この設計では常に同じ計算式を使う。トレンド推定の信頼度（`none`/`low`/`medium`/`high`、測定日数・期間で判定）は参考情報として表示するのみで、計算ロジックの分岐には使わない。

**実装ファイル**:
- `lib/forecast.ts`（新規）: `estimateWeightTrend`（体重トレンド推定）、`estimateAdaptiveTdee`（アダプティブTDEE）、`computeDailyTarget`（目標摂取カロリー・日次目標黒字の単一ソース）、`buildTrajectory`（軌跡予測）、`evaluateOnTrack`（オントラック判定）。
- `app/page.tsx`: 旧OLS回帰・`MIN_DAYS`/`MIN_SPAN`/`calibrated`/`kVal`/`kInfo`を削除し、上記関数群を使用。
- `components/ForecastTab.tsx`: `TrajectoryChart`が`buildTrajectory`を使用。旧「体重×累積黒字」散布図カード（回帰でkを算出する仕組みの表示）は、kが固定値になったことで存在意義が無くなったため削除し、代わりに「ペース診断」カード（申告摂取量・アダプティブTDEE・Google Health推定消費の3値比較、過小申告の兆候を示唆）を新設。
- `lib/advisor.ts`: `AdviceContext`から`k`を削除。AIアドバイザーのプロンプトは新しい実測ペース vs 目標ペースの表現・アダプティブTDEEベースの目標摂取カロリーを使用。
- `lib/advice-core.ts`: `DEFAULT_K`を削除し`lib/forecast.ts`の`K_CONST`に統一。データ取得窓を14日→21日（`TDEE_WINDOW_DAYS`）に拡大。
- `components/SettingsTab.tsx`: 独自に再計算していた`dailyTarget`相当の値（`7200`ハードコード、`app/page.tsx`の値と食い違いうる重複バグ）を削除し、`TabProps`の`dailyTarget`を直接使用するよう統一。

**onTrack判定の変更**: 従来`today.d >= dailyTarget*0.8`（単日の自己申告収支ベース、自己申告バイアスの影響を受ける）だったオントラック判定を、`evaluateOnTrack`（体重トレンドの実ペース vs 目標ペース）に変更した。旧ロジックは`ForecastTab`のツールチップが説明する「予測 vs 実測」という概念とそもそも整合していなかった。
